This contains the most current information about how to use the Solo* APIs

All `POST/PATCH` requests must have content-type: `application/json`

Server Environments and Information

  • Check out URLs page for the current urls for the APIs and Dashboards on different enviornments

  • The API uses JSONAPI for almost all endpoints. The documentation will specify if an endpoint does not conform this protocol.

  • When requesting an object, some endpoints are wired to have associations with other classes. This allows you to include those nested or related objects in the query.

For example:

/api/v1/events/code/123456?include=feedback_survey,feedback_survey.survey_sections,feedback_survey.survey_sections.survey_questions

Here you are requesting an event by code and including the associated feedback_survey with all of the nest survey_questions. The documentation will list the available objects that can be included if it's useful for that functionality.

  • The server current expects two headers app-version and app-platform. The platform is ios or android and the version is a 3 part string of numbers, currently 0.15.0, 0.19.1 respectively.

  • To update this go to /configuration_options and change the app version for the option with name: minimum_app_version and the corresponding platform.

Requesting Limited Models

If you don't want all of the information from an API model, you can use spare fieldsets.

To do that follow this pattern:

`/api/v1/plural_model/:id?fields[plural_model]=attribute1,attribute2`

This will return only those two attributes in the response.

Authentication

In order to access the app, you will need a client token. This allows you to submit the required information and receive a user token and access the full functionality of the application. We use Doorkeeper to implement OAuth 2.0.

Client Tokens

Request a client token using the provided client id and client secret. These tokens expire every 2 hours for security. If you need these client values, contact Logan.

OAuth 2 Client Credentials:

curl -F grant_type=client_credentials
-F client_id=actual_client_id
-F client_secret=actual_client_secret
-X POST "http://url.com/oauth/token"

User Token

Payload

These tokens expire in 2 hours, after the created_at unix time stamp, which is in seconds. You can test them out here: unix time converter

application/x-www-form-urlencoded

Survey Questions

These are the join between a feedback question and a feedback survey, which is why you need to reference the question_id attribute, not the id of the survey_question. Some question types are answered by default when required. That means that the initial value of the input could be their choice, so the user doesn't need to interact with it in order to mark it as answered so they can submit their response. These question types will be indicated.

Attributes

Parameter Type Description
survey_section_id integer
feedback_question_id integer
order integer the order the questions are displayed in the section
question string
kind string
values array an array of values for the labels and input.
default_value string this is only used on the binary and slider types, for now, it is the initial value that the input should be set to when rendering. It will be the number * for the slider and the text for the binary type. Do a case-insensitive comparison.
required boolean the question is required for submission
question_id integer the id of the question this object represents -- USE THIS AS THE feedback_question id when submitting feedback_answers
special If this is a question that renders a custom interface screen instead of a form * input
special_name The name of the screen that it renders

Types of Questions

The allowed types are "text", "binary", "dropdown", "slider", "multi"

TEXT This is a freeform text box, here the value attribute is not used. kind: "text" Submit: The body from the text

BINARY This is represented by a toggle switch with the labels corresponding to the values in the array. Right now this is assumed to be a yes/no question. Answered by default kind: "binary" values: ["Yes", "No"] default_value": "Yes" Submit: Yes or No

DROPDOWN This is a dropdown selector with the array of values given as the options. Only one option can be chosen. "kind": "dropdown" "values": [ "Red Dawn", "Blue Lagoon", "Quiet Skies", "Evening Rain" ] Submit: The string of the selected value

SLIDER This is a slider component, where the scale for everyone is 1-100. The values given are the labels for the left and right sides of the slider, respectively. Answered by default "kind": "slider" "values": ["1", "10"] default_value": "1"

Submit: the integer of the value of the slider

MULTI This is a multiple-select question type where each answer is a checkbox, users are allowed to select all or some of the answers, they must pick at least one. kind: "multi" values: ["Cheetos", "Ritz", "IPA", "Mango"] Submit: A comma-separated string of the selected answers

Special Types of Questions

The current types are: 'focus', 'mood-state', 'energy', 'confidence'. * Focus is the eye chart. * Mood state is the diarama with different characters that switch scenes. * Energy is the image of stick figures. * Confidence is the scene with the cave, rainbow, and mountain.

Feedback Response

A feedback response is a wrapper object for all of the feedback answers a user has for each question in a feedback survey. This allows you to post all of the answers at once, think of it like a wrapper object. A feedback response is to feedback answers like a feedback survey is to survey questions.

Attributes

  • created_at - the datetime the feedback response was created. Don't submit this.

Get Feedback Responses for User

GET /api/v1/feedback_responses?include=effect_persona,variant,feedback_survey&page[size]=5&page[number]=1

Additional Filter &filter[survey_type]=default_product

Here the page size is the number of items you want returned and the page number is the number of the page of items you want. Add any of the listed relationships in the payload into the include section to add them to the response or remove them if they are unneeded.

You can add the filter if you want to only return responses for the product feedback surveys. Allowed values are 'brand', 'event', 'default_product'.

Create a Feedback Response

POST /api/v1/feedback_responses?include=effect_persona,geolocation,feedback_answers

This payload can be confusing so let's break it down into pieces * Responses belong to a person, a survey, and potentially an event or variant. Because the user is logged in, the person is automatically associated on submission, we just need to add the id of the event that it's in response to. Look at the event key in the relationships section as an example. * We have many answers that we want to associate with a given response for the user. Each answer needs to be included in the relationships section with a made up temp-id. Then in the included section, you include each feedback_answer with the temp-id, answer, and the relationship with a feedback_question. * Use the method keys you see below. The update is to make an association. create is to create a new object. * An event or feedback_survey is required. If only an event is provided, then the feedback survey will be automatically added from that. If the feedback_survey is provided it will associated directly. The event if optional if the feedback_survey is given. h * The variant is also optional, but required for product feedback so we know which product the user is providing feedback for.

Basically, just use this example and do the following: 1. Change the event id -- make sure to use the id and not the code for the event or omit if it's not for an event 2. The question is what the users are answering. The left_prompt and right_prompt are the labels for their respective sides under the arc slider, they represent the range of answers for the arc. The rank is the order for the questions.

Baseline Question Attributes

  • right_prompt : string
  • left_prompt : string
  • rank : integer - the order the questions should be shown in
  • fun_fact : object - currently this is used for processing the results of the question, no needed for the app

Error Reports

When the user indicates that there is an issue with a scanned serial number then you submit an error report.

Attributes

  • believe_to_be_counterfeit: a boolean representing whether or not the user believes the item to be counterfeit
  • date_purchased: date the user purchased the product
  • name_of_vendor: name of the person/place where they bought it
  • city_purchased: city where they bought it
  • state_purchased: state where they bought it
  • brand_name: name of the brand of the product they got
  • product_name: name of the product they bought
  • notes: any other notes

Relationships

  • scan: id of the scan object your are reporting about
  • serial_number: string the serial number of the sticker you scanned
  • person: you don't have to explicitly include this, it's pulled from the user token which is REQUIRED
  • geolocation: The geolocation is where the location data is stored for the scan. Put the method create for it and just make the temp id match in both locations, the value is irrelevant. If you are submitting multiple values to be created in the same call then they need to be unique, but if there is only 1 per post it doesn't matter. The geolocation should always be sent if possible
  • purchase location: The purchase location is where the location the use indicates they bought the product.

Submit in pieces

You can submit activation scans in pieces, you can submit the start_raw_value + mj_plants or a batch & variant in the first one. Then submit the end_raw_value in an update.

To get pending or half finished activation scans

GET /api/v1/activation_scans?filter[status]=pending

  • gender: string - male or female
  • birthday: datetime - get the age in years and months and convert to a date relative to today
